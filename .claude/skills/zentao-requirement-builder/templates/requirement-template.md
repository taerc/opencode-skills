## 二、详细内容

### 2.1 进入路径
**用户从哪个页面点击什么按钮进入该功能**

示例：
> 用户在登录页点击"忘记密码"按钮，跳转至手机号验证页。

### 2.2 交互逻辑
**描述具体的业务流程**

#### 操作流程
1. 用户在登录页点击"忘记密码"按钮
2. 系统跳转至手机号验证页
3. 用户输入注册手机号
4. 用户点击"获取验证码"按钮
5. 系统发送短信验证码
6. 用户输入验证码
7. 系统验证通过后跳转至设置新密码页
8. 用户输入新密码
9. 系统保存新密码并跳转至登录页

#### 页面元素
**列出页面上的输入框、按钮、提示文字等**

| 页面 | 元素名称 | 元素类型 | 是否必填 | 说明 |
|------|---------|---------|---------|------|
| 手机号验证页 | 手机号输入框 | 文本输入框 | 是 | 仅支持中国大陆手机号，11位数字 |
| 手机号验证页 | 获取验证码按钮 | 按钮 | - | 点击后60秒倒计时 |
| 手机号验证页 | 验证码输入框 | 文本输入框 | 是 | 6位数字 |
| 手机号验证页 | 下一步按钮 | 按钮 | - | 验证通过后跳转至设置新密码页 |
| 设置新密码页 | 新密码输入框 | 密码输入框 | 是 | 8-16位，必须包含字母和数字 |
| 设置新密码页 | 确认密码输入框 | 密码输入框 | 是 | 与新密码一致 |
| 设置新密码页 | 确认按钮 | 按钮 | - | 保存新密码并跳转至登录页 |

### 2.3 规则限制
**列出业务规则和限制条件**

#### 输入规则
- 手机号：仅支持中国大陆手机号，格式为11位数字
- 验证码：6位数字，有效期5分钟
- 新密码：8-16位，必须包含字母和数字

#### 业务规则
- 短信验证码每天最多发送5次
- 验证码输入错误超过3次，需重新获取
- 新密码不能与旧密码相同
- 同一手机号24小时内最多重置密码3次

#### 异常处理
- 手机号未注册：提示"该手机号未注册，请先注册"
- 验证码错误：提示"验证码错误，请重新输入"
- 验证码过期：提示"验证码已过期，请重新获取"
- 新密码格式错误：提示"密码必须为8-16位，包含字母和数字"
- 两次密码不一致：提示"两次输入的密码不一致"

## 三、验收标准

**格式建议**：使用 Markdown 的复选框列表 `- [ ]`

### 3.1 功能验收

- [ ] 在登录页点击"忘记密码"，跳转至手机号验证页
- [ ] 输入正确的手机号，点击"获取验证码"，按钮变为60秒倒计时，不可重复点击
- [ ] 输入错误的验证码（过期、错误），页面提示"验证码错误"
- [ ] 输入正确验证码后，跳转至设置新密码页
- [ ] 新密码必须为8-16位，包含字母和数字
- [ ] 两次密码不一致时，提示"两次输入的密码不一致"
- [ ] 设置新密码成功后，跳转至登录页，可以使用新密码登录

### 3.2 异常验收

- [ ] 手机号未注册时，提示"该手机号未注册，请先注册"
- [ ] 验证码发送超过5次/天，提示"今日验证码发送次数已达上限，请明天再试"
- [ ] 验证码输入错误超过3次，提示"验证码错误次数过多，请重新获取"
- [ ] 新密码与旧密码相同时，提示"新密码不能与旧密码相同"
- [ ] 24小时内重置密码超过3次，提示"操作过于频繁，请明天再试"

### 3.3 性能验收

- [ ] 短信验证码发送成功，延迟不超过3秒
- [ ] 页面加载时间不超过2秒
- [ ] 验证码验证响应时间不超过1秒

### 8.2 常见问题

**Q1：为什么限制每天只能发送5次验证码？**
A：防止恶意刷短信验证码，控制短信成本。

**Q2：为什么验证码有效期是5分钟？**
A：平衡安全性和用户体验，5分钟足够用户完成操作，同时降低验证码被泄露的风险。

**Q3：为什么新密码不能与旧密码相同？**
A：提升账户安全性，防止用户设置过于简单的密码。

## 使用说明

### 模板填写指南

1. **需求标题**：必须符合"角色 + 想要什么 + 以便于"格式
2. **背景与动机**：描述清楚用户在什么场景下遇到什么问题，为什么需要这个功能
3. **详细内容**：描述具体的业务流程、交互逻辑、规则限制，越详细越好
4. **验收标准**：必须量化、无歧义，使用复选框列表格式
5. **附加信息**：埋点、日志、兼容性等可选信息

### 常见误区

| 错误类型 | 错误示例 | 正确写法 |
|---------|---------|---------|
| 过于抽象 | "优化报表查询速度" | "将月度销售报表的查询加载时间，从现在的5秒降低至1秒以内" |
| 需求过大 | "开发一套全新的CRM系统" | 拆分为：客户管理、商机管理、合同管理等多个需求 |
| 解决方案先行 | "在数据库增加一个type字段" | "需要在后台用户列表，能够区分哪些是'普通会员'，哪些是'VIP会员'" |

### 自检清单

在保存需求文档之前，请检查：

- [ ] **角色明确**：谁会用？（普通用户？管理员？）
- [ ] **动作清晰**：具体要操作什么？（点击？输入？查看？）
- [ ] **价值存在**：为什么要做？（增加收入？减少客诉？提升效率？）
- [ ] **边界清楚**：什么情况能做？什么情况不能做？（规则和限制写清楚了吗？）
- [ ] **验收可量化**：测试人员能直接复制你的验收标准去写测试用例吗？
