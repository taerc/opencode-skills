## 6. 变更管理

### 6.1 变更流程

```
1. 变更申请（提交变更需求）
   ↓
2. 影响评估（产品、技术、时间、成本）
   ↓
3. 变更审批（产品负责人、技术负责人）
   ↓
4. 变更实施（更新文档、调整计划）
   ↓
5. 变更通知（通知所有相关方）
```

## 填写指南

### 各章节填写说明

| 章节 | 填写说明 | 注意事项 |
|------|----------|----------|
| 1. 项目信息 | 基本信息，便于识别和追踪 | 确保信息准确 |
| 2. 交付范围 | 核心内容，明确交付什么、不交付什么 | 范围要具体，防止歧义 |
| 3. 验收标准 | 明确的完成标准 | 标准要可衡量、可测试 |
| 4. 依赖清单 | 识别内外部依赖 | 提前确认依赖方计划 |
| 5. 风险与应对 | 识别风险并制定应对措施 | 风险要有触发条件和应对动作 |
| 6. 变更管理 | 明确变更流程 | 严格执行变更管理，防止范围蔓延 |
| 7. 沟通计划 | 确保信息同步 | 沟通要及时、透明 |
| 8. 发布计划 | 确保顺利发布 | 发布检查清单不能省略 |

### 常见错误

❌ **错误示例**：
- 范围描述模糊："优化用户体验"
- 验收标准不可衡量："性能要好"
- 依赖项未识别或状态不明确
- 风险未识别或未制定应对措施
- 变更流程不明确或执行不严格

✅ **正确示例**：
- 范围描述具体："将首页加载时间从 3 秒降至 1 秒"
- 验收标准可衡量："P95 响应时间 < 500ms"
- 依赖项有负责方、交付时间、状态
- 风险有等级、触发条件、应对措施
- 变更流程明确，严格执行
