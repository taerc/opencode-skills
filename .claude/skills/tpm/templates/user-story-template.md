## 用户故事编写规范

### 标准格式

```markdown
**用户故事 {编号}：{故事名称}**
> **作为** {角色}，**我希望** {功能}，**以便** {价值}。

**详细需求**：
- 需求点1
- 需求点2

**核心指标**：
- 指标1
- 指标2

**业务价值**：业务价值说明。
```

### 编号规则

| 分类 | 编号前缀 | 示例 |
|-----|---------|------|
| 日报 | D | D1, D2, D3... |
| 周报 | W | W1, W2, W3... |
| 月报 | M | M1, M2, M3... |
| 年报 | Y | Y1, Y2, Y3... |
| 风险管理 | R | R1, R2, R3... |
| 考核评价 | K | K1, K2, K3... |
| 其他 | 按功能命名 | Task-1, Defect-1... |

### 优先级定义

| 优先级 | 含义 | 说明 |
|-------|------|------|
| P0 | 必需 | MVP核心功能，第一期必须实现 |
| P1 | 重要 | 核心功能增强，第二期实现 |
| P2 | 次要 | 优化功能，第三期实现 |

### 编写要点

1. **角色明确**：清楚定义谁在使用这个功能
2. **功能具体**：不要写模糊的功能描述
3. **价值清晰**：说明为什么要做这个功能
4. **需求可执行**：详细需求要让开发能直接实现
5. **指标可量化**：核心指标要有具体数字或百分比

### 常见错误

| 错误类型 | 错误示例 | 正确写法 |
|---------|---------|---------|
| 角色模糊 | "用户希望..." | "配网专责希望..." |
| 功能抽象 | "我希望优化体验" | "我希望在首页直接看到今日任务完成率" |
| 价值缺失 | "...以便完成任务" | "...以便快速掌握当日工作进展，及时汇报" |
| 指标模糊 | "提高效率" | "任务完成率达到95%以上" |

## 示例：完整的用户故事

**用户故事 D1：每日工作总结**
> **作为** 配网专责，**我希望** 每天下班前能快速生成当日工作总结报告，**以便** 向上级汇报工作进展，及时发现当日异常情况。

**详细需求**：
- 今日任务执行情况汇总（已完成/执行中/待执行）
- 今日航线统计（数量、里程、覆盖杆塔数）
- 今日图片统计（上传数、审核数、待审核数）
- 今日缺陷发现统计（新增缺陷数、处理缺陷数）
- 今日异常问题汇总（延期任务、失败任务、异常中止）
- 明日计划任务预览

**核心指标**：
- 任务完成率
- 图片上传率
- 缺陷新增数
- 异常任务数

**业务价值**：支撑日常工作汇报，及时发现异常。

## 使用说明

### 模板填写指南

1. **文档信息**：填写版本、日期、作者等基本信息
2. **背景**：描述业务背景和核心价值
3. **分类总览**：按维度分类统计用户故事
4. **详细说明**：按标准格式编写每个用户故事
5. **优先级矩阵**：按P0/P1/P2分类排列
6. **核心字段**：汇总所有用户故事需要的数据库字段
7. **实施建议**：分阶段实施计划

### 自检清单

在保存用户故事文档之前，请检查：

- [ ] **编号正确**：编号按分类顺序排列，无重复
- [ ] **角色明确**：每个故事都清楚定义了使用者角色
- [ ] **功能具体**：功能描述具体可执行，不模糊
- [ ] **价值存在**：每个故事都有清晰的业务价值
- [ ] **需求完整**：详细需求覆盖核心功能点（3-6条）
- [ ] **指标量化**：核心指标可度量，有具体数值或百分比
- [ ] **优先级合理**：P0/P1/P2分布合理
